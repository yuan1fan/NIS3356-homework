# 数据流建议

## 模块职责

当前建议的数据流如下：

1. **数据爬取模块**
   - 获取微博热搜词条及对应帖子。
   - 提供帖子正文、图片、视频、音频等原始来源。
   - 保存热搜排名、热度、互动指标、来源链接等元数据。

2. **ASR 模块**
   - 处理爬虫提供的音频或视频文件。
   - 提取其中的语音并生成转写文本。
   - 可附带时间戳、置信度、脱敏结果和安全标签。

3. **图像处理模块**
   - 处理爬虫提供的图片或从视频中提取的图像。
   - 输出 OCR 文本或包含文本块、置信度、区域信息的结构化结果。

4. **NLP 模块**
   - 处理帖子正文、评论等原始文本。
   - 输出清洗、分词、实体识别或其他文本处理结果。

5. **数据分析模块**
   - 原则上接收 ASR、图像处理和 NLP 三个处理模块转换后的统一 JSONL。
   - 不直接导入其他模块内部代码，也不依赖其内部 Python 数据结构。
   - 以稳定的文件格式作为模块边界，便于独立测试和替换实现。

```text
数据爬取
  ├─ 音频/视频 ─> ASR ─────────┐
  ├─ 图片 ─────> 图像处理 ─────┼─> 统一 JSONL ─> 数据分析
  └─ 正文/评论 ─> NLP ─────────┘
```

## 爬虫数据的定位

爬虫输出是上游原始数据，不建议作为数据分析模块的最终固定接口。

- 爬虫正文可以作为临时测试数据，帮助验证统计、情感分析或主题聚类流程。
- 热搜排名、热度、互动指标、媒体 URL 和来源链接适合写入统一记录的
  `metadata`。
- 如果 ASR、图像处理或 NLP 模块尚未完成，数据分析模块可以临时直接读取
  爬虫正文做 Demo。
- 临时读取爬虫正文不代表最终耦合方式。正式流程仍应由对应处理模块完成
  清洗、识别或结构化，再输出统一 JSONL。

## 统一 JSONL

当前统一输入格式保持不变。文件使用 UTF-8 编码，每行一条 JSON 记录：

```json
{
  "document_id": "唯一编号",
  "source_type": "asr/ocr/nlp/manual",
  "source_path": "来源文件路径",
  "text": "用于分析的文本",
  "raw_text": "原始文本，可选",
  "metadata": {}
}
```

约束保持如下：

- `document_id` 和 `text` 必填。
- `source_type`、`source_path`、`raw_text` 和 `metadata` 可选。
- 数据分析默认只处理 `text`。
- 原始内容、置信度、排名、热度、时间戳和媒体来源等信息放入
  `raw_text` 或 `metadata`。
- 各处理模块可独立生成 JSONL，之后再合并分析，不要求直接调用彼此的
  内部函数。

## 当前边界

现阶段不实现爬虫结果转换脚本，也不修改数据爬取、ASR、图像处理或 NLP
模块。待上游输出格式稳定后，再分别设计面向文件的转换或汇总流程。
